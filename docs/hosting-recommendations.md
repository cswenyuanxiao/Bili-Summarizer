# Bili-Summarizer 免费部署方案推荐

针对目前 Render 15分钟无操作后休眠（冷启动）的问题，我们调研了国内外多个免费方案。以下是针对你项目的最佳推荐：

## 🏆 综合推荐

| 方案 | 适用场景 | 休眠策略 | 推荐指数 | 关键优势 |
| :--- | :--- | :--- | :--- | :--- |
| **Hugging Face Spaces** | **AI 应用/Demo** | **48小时** 无操作后休眠 | ⭐⭐⭐⭐⭐ | 专为AI设计，休眠由于时间长，体感几乎是"Always On" |
| **Render + Uptime** | **现有方案改良** | **永不休眠** (需配置) | ⭐⭐⭐⭐ | 成本最低的改动，利用 750小时免费额度 |
| **Oracle Cloud** | **极客/运维** | **永不休眠** (真VPS) | ⭐⭐⭐ | 性能最强 (4核24G内存)，但注册极其困难 |

## 方案 1: Hugging Face Spaces (强烈推荐) 🚀

Hugging Face Spaces 专为 AI 应用托管设计，支持 Docker（可以用我们现有的 Dockerfile）。

- **休眠机制**: 免费层级 (CPU Basic) 在 **48小时** 无流量后才会暂停。这意味着只要每两天有人访问一次，它就一直在线。
- **优点**:
  - 不需要像 Render 那样频繁冷启动。
  - 即使休眠，唤醒速度通常比 Render 快。
  - 完全免费，不仅支持 HTTP，还天然支持 Gradio/Streamlit/FastAPI。
  - 没有任何 B站 反爬虫风险（IP 信誉好）。
- **迁移难点**: 极低。只需上传 Dockerfile 和代码即可。

## 方案 2: Render + Uptime Robot (最快修补) ⚡

如果你不想迁移，可以通过外部监控服务"伪造"流量，防止 Render 休眠。

- **原理**: Render 免费层级每月有 **750小时** 运行额度。
  - `24小时 × 31天 = 744小时`
  - 如果你的账号下 **只有这一个服务**，你可以让它 24/7 运行而不超额。
- **操作步骤**:
  1. 注册 [Uptime Robot](https://uptimerobot.com/) (免费)。
  2. 创建一个新的 HTTP Monitor。
  3. URL 填你的 Render 网址 (如 `https://bili-summarizer.onrender.com/health`)。
  4. 间隔设为 **5分钟** (Render 休眠是15分钟，所以5分钟一次心跳足够)。
- **风险**:
  - 如果你在同一个 Render 账号下部署了数据库或其他服务，总时间会超过 750小时，导致月底服务被强制暂停。
  - 这是一个"灰色地带"用法，但目前广泛被使用。

## 方案 3: Oracle Cloud Always Free (硬核之选) 🐢

甲骨文云提供业内最慷慨的永久免费层级。

- **配置**: VM.Standard.A1.Flex (ARM 架构), 4 OCPU, 24GB RAM。
- **优点**: 这是一台真正的 VPS，完全受你控制，永不休眠，性能极强。
- **缺点**:
  - **注册门槛极高**: 中国信用卡极难通过验证（玄学）。
  - **资源紧缺**: 热门区域（东京、圣何塞、新加坡）常年开不出机器 ("Out of Capacity")。
  - **维护成本**: 需要自己维护 Linux 系统安全。
- **建议**: 如果你碰巧能注册下来，这是绝对的首选。

---

## ❌ 不推荐的方案

- **Zeabur**: 体验很好，但免费额度 ($5/月) 是按资源使用量扣费的。如果要 "Always On" (24/7 运行)，Docker 容器的内存消耗很可能会迅速耗尽额度，导致服务暂停。适合开发测试，不适合长期挂机。
- **Fly.io**: 已取消免费额度，现在必须绑定信用卡且按量付费，虽然很便宜，但不是"免费"。
- **国内云厂商 (阿里云/腾讯云)**: 只有短期试用（1-3个月），且 web 服务必须备案（ICP），对个人项目过于繁琐。

## 💡 总结建议

1. **短期/立刻解决**: 对你的 Render 服务设置 **Uptime Robot** 监控（5分钟一次）。这是零成本且立刻生效的方案。
2. **长期/最佳体验**: 迁移到 **Hugging Face Spaces**。48小时的休眠阈值对用户极其友好，基本感觉不到冷启动。
